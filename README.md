# paladin
项目是开发独立游戏《沧澜求仙录》时，为最大程度提升开发效率，解耦策划/前端/后端/美术工作与开发完成。
以实用主义为核心，一切的目标是提升开发效率。

# 设计思路
以xlsx为核心，xlsx记录的数据和关系是一切的基础
通过解析工具产出策划数据(json)，通过代码生成产出unity和go服务器桩函数(*.go, *.cs)
生成数据文件，类型定义和数据读取完全代码生成，避免策划频繁修改时前后端不同步
以一种对策划不友好的方式定义表头，策划改数据很清晰，但改结构十分苦难，改结构几乎必然涉及到需求变更，应该在开发的协助下进行

# 产出
策划数据文件 data/*.json
go桩函数 默认是 *.dbc.go，默认package dbc（传承自wow o_0?）
cs桩函数 默认是 *.cs

## 特性
1. enum.xlsx 枚举转换表，将策划表中的可读字符串转换成数字。（比如 冰冻特效，对应的effectId是5，但5没有可读性）
2. 可选择对xlsx的子表选择性解析。比如装备子类：刀，剑，鞋，从策划角度，他们放在不同子表中更清晰，但程序可能更愿意统一结构存储；另一方面，对于活动/任务，策划希望把所有相关内容放一张工作薄中，但程序会有多种实现结构，因此需要对子表分别解析这个灵活性是非常重要的。
3. 能根据xlsx生成前端(c#)/后端(go)的桩文件，数据文件，类型定义和数据读取完全代码生成，极大减少工作量和复杂度。
4. 可指定行解析/列解析，主要用于全局配置表，全局配置表按行配置数据更好。
5. 由一个统一的config.toml管理解析规则（枚举关系，输出name，解析xlsx表等等）
6. 便于调试的解析系统，尽可能多尽可能详细的汇报策划表配置错误，使用中总结调试成本极低。
7. （部分实现）参数展开，可将{0}, {1}这样的参数替换成参数表中内容。
8. （部分实现）多语言支持，水平扩展，生成到客户端文件夹locale/cn, en, kr这些文件夹下。客户端可根据设置加载语言包

## 局限性
1. 不支持slice和map的多级结构. 只能使用简单数据结构或单层struct（实际使用中已经足够了，作为数据没有更复杂结构的需求）。
2. 多层struct是允许的，但不允许子struct内部再slice或map，建议提取到外层，不然xlsx表可读性太差，xlsx可读/高效才是第一位的。
3. 做slice和map时，要求使用的名称为单一字符串如[awake]或驼峰字符串如[awakeMaterial]，不允许snake字符串，不然解析会报错。

## 表达式
* \<struct> 尖括号用于表达子struct
* [slice] 方括号用于表达slice
* {map} 大括号用于表达map
* \- 横杠用于略去解析（多用于备注，说明表单内容，但不输出）
* 同一slice/map的不同member，使用[slice]#1 字段描述，比如slice类型有type和value两个字段，则这两个字段的desc栏都填[slice]#1, 第二个value的desc栏填[slice]#2
